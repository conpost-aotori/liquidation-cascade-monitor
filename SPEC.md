# 偏りスコア 実装仕様書 — 清算カスケード方向予測モジュール

> BTC清算予兆モニターに「どちらのポジションが踏まれやすいか」を判定する偏りスコア機能を追加する仕様。
> 正本ロジックは `src/liqmap/bias.py`、検証は `scripts/test_bias.py`（本書 §10 のベクター）。

---

## 1. 目的とコンセプト
清算カスケード(連鎖清算)は、価格がある水準に達して強制清算が発生し、それがさらに価格を動かして次の清算を呼ぶフィードバックループ。本モジュールは2軸で「鳴らすべきか」を判定する。
- **軸① 偏りスコア**: いま **どちらの side が過密で脆いか**(=踏まれやすい側)を −100〜+100 で定量化
- **軸② カスケード近接度ゲート**: その引き金となる清算クラスターが **価格的に近いか**(=いつ起きそうか)

両方が揃ったときだけアラート発火。単体シグナルを避け誤検知を潰す設計。

### 符号の規約(最重要・全体で統一)
```
+100 ← ショートが過密 → 上方向スクイーズ(踏み上げ)リスク … 緑/上
   0 ← 中立
-100 ← ロングが過密 → 下方向カスケードリスク         … 赤/下
```
ダッシュボードの配色(赤=ロング、緑=ショート)と一致させる。

---

## 2. 軸① 偏りスコアの構成
4要素の重み付き合成。各要素は符号付きで合算し整数に丸める。

| # | 要素 | 重み | 役割 |
|---|------|-----|------|
| C1 | ファンディング | 40 | 払っている側=過密で脆い側 |
| C2 | OI速度 × 価格方向 | 25 | その過密が「いま積み上がっているか」 |
| C3 | クラスター偏り | 20 | 清算想定額の上下非対称(ポジション分布) |
| C4 | スマートマネー乖離 | 15 | 勝ち組の建玉がリテールと逆か(Nansen) |

## 3. 軸② カスケード近接度ゲート
- スコアの符号で脆い側の引き金方向を決める(score<0→価格より下のロング、score>0→上のショート)
- 「大口(dense)」: notional ≥ `max(GATE_DENSITY × 同方向合計, GATE_ABS_FLOOR)`
- **ゲート open**: 最寄り dense クラスターまでの距離 ≤ `GATE_DIST`(2%)

## 4. ステート機械(発火条件)
| ステート | 条件 |
|---------|------|
| 発火 | `|score| ≥ 60` かつ ゲート open |
| 監視 | `|score| ≥ 40` または ゲート open |
| 静観 | 上記以外 |

bot が X に投稿するのは発火のときだけ(本リポでは現状そのまま全枠投稿。ゲート活用は今後)。

| 入力 | 取得元 | 状態 |

| `clusters`, `*_total` | 既存の清算推定パイプライン | ✅ 実装済 |
| `smart_money_net` | HL勝ち組(allTime PnL上位20%)のBTCネット | ✅ 実装（HLネイティブ代理。Nansen不要） |

> 自動bot(GitHub Actions)は **C1+C3+C4 をライブ算出**（C4=HL勝ち組ベースの代理指標）。**C2(OI速度)はOI時系列ストア**（毎回OIを保存・Actionsキャッシュで永続化）で、約24hぶん貯まると有効化（それまでパネルの「未反映」に表示）。Nansenを使う場合は C4 を差し替え可能。

## 7. 発火時の出力(X ポスト テンプレート)
方向ラベル(score>0→ショート過熱/上踏み、score<0→ロング過熱/下落カスケード)、偏りスコア、各要素内訳、引き金価格を明示。

## 8. バックテスト用ログ
発火ごとに `timestamp, coin, score, gate, state, price` を保存し、別ジョブで +24h/+72h 後の価格・的中を追記。

## 9. 実装アーキテクチャ
Python。HL+Nansen取得 → `bias_score` → `cascade_gate` → `state` → 発火かつ非重複なら投稿。

## 11. 較正の指針
- magic number は `bias.py` 冒頭に集約。既存履歴でバックテストして調整。
- 発火頻度は絞る(鳴ったら本物)。
- C3(上下非対称=過密度)とゲート(最寄り距離=タイミング)は同じクラスターデータを別の意味で使う。混同しない。
